# ゴールまでのタスク一覧

**ゴール**: GitHub Actions で定期実行され、新着物件が自分のメールアドレスに届く状態にする。

---

## 1. 前提・準備

| # | タスク | 内容 | 備考 |
|---|--------|------|------|
| 1.1 | GitHub アカウント作成 | まだ持っていない場合、[github.com](https://github.com) でサインアップ | 無料プランで可 |
| 1.2 | リポジトリの用意 | このプロジェクトを GitHub にプッシュするためのリポジトリを作成 | プライベート or パブリック。プライベートなら月2,000分の無料枠 |
| 1.3 | メール送信用の準備 | 送信に使うメールアドレスを決め、Gmail なら「アプリパスワード」を発行 | Gmail: 設定 → セキュリティ → 2段階認証 → アプリパスワード |

---

## 2. バッチの実装（設計書に沿って作る）

| # | タスク | 内容 | 備考 |
|---|--------|------|------|
| 2.1 | スクレイピングの分離 | `app_realestate.py` から戸建・中古マンション用のスクレイピングを `batch/scraper.py` などに分離 | 既存コードを流用 |
| 2.2 | 所要時間マスタ読み込み | `station_time.xlsx` を読む `batch/station_time.py` を実装 | pandas + openpyxl |
| 2.3 | 既知物件一覧の読み書き | `known_properties.json` の読み込み・更新・重複判定を `batch/storage.py` に実装 | 新着判定・重複判定ロジック含む |
| 2.4 | 抽出条件スコアリング | `extraction_conditions.yaml` を読み、スコア計算する `batch/extractor.py` を実装 | 閾値以上のみ対象 |
| 2.5 | メール送信 | 新着一覧を送る `batch/mailer.py` を実装。**メール本文に「全物件一覧（条件ポイント降順）」へのリンクを追加** | smtplib。Gmail 等対応 |
| 2.6 | 全物件一覧ページの生成 | **今回の検索で取得した物件のうち閾値以上のもの**を条件ポイント降順で並べた `docs/properties_list.html` を毎回生成。データソースは既知一覧ではなく当該実行のスクレイピング結果（死にリンク防止）。既知一覧には `extraction_score` も保存 | GitHub Pages で公開する場合はリポジトリで Pages を有効化 |
| 2.7 | バッチのオーケストレーション | 上記を組み合わせる `run_batch.py` を実装 | 設定は環境変数 or config |

## 3. 手動で動くかテストする（ローカル）

| # | タスク | 内容 | 備考 |
|---|--------|------|------|
| 3.1 | 入力ファイルの準備 | **検索用URLリスト**は `docs/suumo_url_list.csv` に配置済みで使用。`station_time.xlsx` はプロジェクト直下に配置 | 設計書のデフォルトは `url_list_path: ./docs/suumo_url_list.csv` |
| 3.2 | 設定の準備 | メール送信の設定（SMTP・アドレス）を `config.yaml` または環境変数で用意 | 本番と同じ Gmail 等でよい |
| 3.3 | ドライラン | `python run_batch.py --dry-run` でスクレイピング～新着判定までが動くか確認 | メールは送らない |
| 3.4 | 1回目の本実行 | `--dry-run` なしで実行。新着が「既知一覧」に追加され、メールが届くか確認 | 初回は全件が新着になる想定 |
| 3.5 | 2回目の本実行 | 再度実行。新着がなくメールが送られない（または既知一覧が更新されない）ことを確認 | 新着検出・重複判定の確認 |

## 4. GitHub に上げて Actions で動かす

| # | タスク | 内容 | 備考 |
|---|--------|------|------|
| 4.1 | Git リポジトリ化 | まだなら `git init`、`.gitignore` の整備 | `config.yaml` はコミットしない（Secrets で渡す） |
| 4.2 | リモートにプッシュ | GitHub のリポジトリをリモートに追加し、`main` 等にプッシュ | `known_properties.json` はコミットする |
| 4.3 | Secrets の登録 | リポジトリの Settings → Secrets and variables → Actions で SMTP_* / FROM_EMAIL / TO_EMAILS を登録 | 値はローカルで使ったものと同じでよい |
| 4.4 | ワークフローの確認 | `.github/workflows/run_batch.yml` がリポジトリに含まれているか確認 | 既に配置済み |
| 4.5 | 手動実行でテスト | GitHub の Actions タブから「Run workflow」で 1 回実行 | ログでエラーがないか、メールが届くか確認 |
| 4.6 | 全物件一覧のリンク先を用意（任意） | メールの「全物件一覧」リンクをクリックできるようにする。**GitHub Pages** を有効にすると `https://<user>.github.io/<repo>/docs/properties_list.html` で表示可能。または raw リンク（`raw.githubusercontent.com/...`）でHTMLを直接開く | Pages: リポジトリ Settings → Pages → Source を GitHub Actions または main の /docs に設定 |
| 4.7 | スケジュール実行の確認 | 翌日、指定時刻（JST 6:00 等）に自動実行されたか確認 | 必要なら cron を調整 |

---

## 進め方の目安

1. **1. 前提・準備** を済ませる（GitHub アカウント、メールのアプリパスワード）
2. **2. バッチの実装** を設計書（`DESIGN_SUUMO_BATCH.md`）に沿って進める
3. **3. 手動テスト** でローカルで「動く」ことを確認してから GitHub に上げる
4. **4. GitHub に上げて Actions で動かす** で Secrets を入れ、手動実行 → スケジュール確認

実装（2）は Phase 1 から順に進めると、早い段階で `--dry-run` による動作確認ができます。
